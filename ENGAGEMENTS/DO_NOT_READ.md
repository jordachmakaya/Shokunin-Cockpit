# DO_NOT_READ.md

## Purpose

This file tells AI coding agents which files and folders must not be read unless explicitly requested by the project owner.

The goal is to reduce token usage, avoid reading generated files, avoid leaking secrets, and keep the agent focused on source code only.

These rules apply to Antigravity and any AI coding agent working on this Nuxt project.

---

# 1. Never read these folders

Do not read, scan, summarize, index, open, or analyze these folders:

```txt
node_modules/
.pnpm-store/
.nuxt/
.output/
.data/
.git/
```

Reason:

- generated or dependency files
- extremely large token cost
- not useful for normal implementation work
- may contain cached/generated/runtime data
- can confuse the agent with non-source files

---

# 2. Never read secret or environment files

Do not read these files:

```txt
.env
.env.*
!.env.example
```

Allowed exception:

```txt
.env.example
```

The agent may read `.env.example` because it should contain placeholders only.

Never read or expose:

```txt
API keys
tokens
GitHub PATs
Dokploy webhooks
CRM credentials
PostHog keys if private
database URLs
private certificates
service secrets
```

If an environment variable is needed, ask the project owner to provide only the variable name or a redacted example.

---

# 3. Never read build artifacts

Do not read:

```txt
.output/
.nuxt/
dist/
coverage/
playwright-report/
test-results/
.cache/
```

Reason:

- generated files
- high token usage
- not source of truth
- should be regenerated by commands

---

# 4. Never read lockfiles unless dependency work requires it

Do not read the full content of:

```txt
pnpm-lock.yaml
package-lock.json
yarn.lock
```

Allowed:

- check if the file exists
- update it through package manager commands
- inspect a small relevant section only when debugging dependency resolution

Do not summarize or scan the whole lockfile.

---

# 5. Avoid reading large media and binary files

Do not read or analyze:

```txt
public/videos/
public/images/
*.mp4
*.mov
*.webm
*.png
*.jpg
*.jpeg
*.webp
*.gif
*.ico
*.pdf
*.zip
```

Allowed:

- reference their paths
- check that they exist
- use them only when the task is explicitly about images, videos, PDFs, or assets

Do not consume tokens inspecting media unless the user explicitly asks.

---

# 6. Source files the agent may read

The agent may read these folders and files when relevant:

```txt
app/
components/
composables/
content/
server/
shared/
test/
tests/
nuxt.config.ts
package.json
tsconfig.json
vitest.config.ts
playwright.config.ts
Dockerfile
.dockerignore
.github/workflows/
README.md
RULES.md
```

The agent should read only the files relevant to the task.

Do not scan the entire repository by default.

---

# 7. Preferred reading order

Before editing code, read in this order:

```txt
1. RULES.md
2. README.md
3. package.json
4. nuxt.config.ts
5. the specific files related to the task
```

For deployment work:

```txt
1. RULES.md
2. Dockerfile
3. .dockerignore
4. .github/workflows/*
5. package.json
```

For Nuxt page/component work:

```txt
1. RULES.md
2. nuxt.config.ts
3. app/app.vue
4. app/pages/*
5. relevant components only
```

For server/API work:

```txt
1. RULES.md
2. server/api/*
3. server/services/*
4. server/utils/*
5. server/config/*
6. shared validation/types if present
```

---

# 8. Token-saving behavior

The agent must:

- read only what is needed
- avoid repository-wide scans
- avoid opening generated folders
- avoid reading ignored files
- avoid reading dependencies
- avoid reading binary files
- ask before inspecting large files
- summarize assumptions before making large edits
- make small, reviewable changes

If a task requires broad context, the agent must first explain which folders it needs to inspect and why.

---

# 9. Git ignored files

Files ignored by `.gitignore` should normally be treated as non-source files.

Do not read ignored files unless explicitly instructed.

Common ignored files include:

```txt
.env
.data/
.nuxt/
.output/
.pnpm-store/
node_modules/
```

---

# 10. Safety rule

If a file may contain secrets, credentials, tokens, private customer data, private CRM records, or local machine configuration, do not open it.

Ask the project owner for a redacted sample instead.

---

# 11. Final instruction

Do not spend tokens on generated files, dependencies, media files, local caches, or secrets.

Focus on:

```txt
source code
configuration
content
tests
deployment files
project rules
```

The source of truth is the editable project code, not generated output.
